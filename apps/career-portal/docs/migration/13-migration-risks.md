# 13 — Migration Risks

## Method

Risks were derived from read-only inspection of the legacy React/Vite frontend, Express API, Mongoose schemas, route middleware, upload/PDF/email services, scripts, and the empty `career-portal` implementation target. Probability and impact use `Low`, `Medium`, `High`, and `Critical`. Blockers are listed separately in the parity matrix and plan; a risk is not itself evidence that implementation has started.

## Highest-priority pre-implementation risks

The following must be reduced before broad implementation:

1. **R-03/R-04** — authorization and session architecture.
2. **R-05/R-06/R-24** — data compatibility, workflow invariants, and rollback authority.
3. **R-07/R-08/R-14/R-26** — sensitive data, uploads, origin security, and credential hygiene.
4. **R-02/R-19** — observed behavior and API contract uncertainty.
5. **R-09/R-10/R-11** — runtime-heavy parsing/PDF/email side effects.

## Risk acceptance requirements

A risk may be accepted only with an owner, expiry/review date, explicit rationale, monitored signal, and approved fallback. “Legacy already behaves this way” is not sufficient justification for carrying a security or privacy defect into the target.
